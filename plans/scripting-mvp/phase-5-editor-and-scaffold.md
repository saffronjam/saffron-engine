# Phase 5 — editor Inspector + project scaffold

**Status:** NOT STARTED

## Goal

Make the per-entity scripting authorable from the editor: render the **`ScriptComponent` in the
Inspector** as an ordered list of script slots (assign a `.lua`, reorder, remove), and render each slot's
**script-declared fields** as widgets that write per-instance overrides. Plus the authoring UX: **New
Project** scaffolds `src/` + a starter example script, and the editor offers **Open in VS Code**. This is
the only phase touching the editor/Tauri layers.

## What exists to build on

- The Inspector renders components from data fetched over the control plane (`InspectorPanel.tsx`); the
  component DTOs come from `@saffron/protocol` (regenerated by `gen.ts`). The `ScriptComponent`/`ScriptSlot`
  shapes (Phase 2) and the `get-script-schema` + override-set commands (Phase 4) are already on the wire.
- Project scaffolding is **engine-side** in `createProject` (`assets.cppm:489-515`): it resolves the root,
  `setAssetRoot(root/"assets")` (`:512`, which `create_directories` the asset tree), then `saveProject`
  (`:514`). `loadProject` (`:448-487`) re-`setAssetRoot`s on open (the ensure-on-load precedent). The module
  already includes `<filesystem>`; `<fstream>` is used elsewhere.
- Tauri: frontend deps are `@tauri-apps/api` + `@tauri-apps/plugin-dialog`; capabilities grant
  `dialog:default` but **no `shell:*`** (`capabilities/default.json`). The store holds `store.project`
  (`root`/`path` from `ProjectInfoDto`, `control_commands_asset.cpp:46-49`); menus are `ProjectMenu.tsx` /
  `MenuBar.tsx`; the New Project surface is `ProjectStartupModal.tsx`.

## Work

### 1. Inspector: the script-slot UI

In the editor Inspector (`InspectorPanel.tsx` + a new `ScriptComponentSection`):

- Render `ScriptComponent.scripts` as an **ordered list of slots** (mirroring how a material set / array
  component renders): per slot show the script path with **assign** (a file picker scoped to the project
  `src/`, via the dialog plugin) / **remove** / **reorder** (drag or up/down — list order is the runtime
  execution order from Phase 2, so reordering is meaningful).
- For each slot, call **`get-script-schema(path)`** (Phase 4) and render one widget per declared field —
  number box / checkbox / text / vec3 (xyz) — pre-filled with the default, showing the current override if
  any. Editing a widget writes the slot's **override** (the Phase-4 set-override path). Reconcile the
  stored overrides against the fetched schema (drop renamed/removed, show new defaults).
- Errors from a broken script (`get-script-schema` returning `Err`, or `drain-script-errors` entries during
  play) surface inline (a small banner / the existing notice mechanism) so authors see the traceback
  without leaving the editor.

### 2. Engine: scaffold `src/` + a starter script on New Project

In `createProject` (`assets.cppm:489-515`), after `setAssetRoot(...)` (`:512`) and before `saveProject`
(`:514`):

```cpp
std::error_code ec;
std::filesystem::create_directories(root / "src", ec);
const auto example = root / "src" / "example.lua";
if (!std::filesystem::exists(example)) { std::ofstream(example) << kStarterScript; }
```

`kStarterScript` is a minimal class with a declared field, so a new project is immediately demonstrable:

```lua
-- example.lua — attach this to an entity's Script component, then press Play.
local Example = {}

Example.properties = {
  speed = 1.0,   -- editable in the Inspector
}

function Example:on_update(dt)
  local x, y, z = self.entity:get_position()
  self.entity:set_position(x, y + self.speed * dt, z)
end

return Example
```

Mirror the `src/` ensure in `loadProject` (`:448-487`, near `setAssetRoot`) so pre-existing projects gain
the folder on open. Idempotent — only write when absent. (The `src/` dir sits beside `assets/`; confirm no
"clean project" routine assumes the project dir holds only `project.json` + `assets/`.)

### 3. Editor: "Open in VS Code"

Use `tauri-plugin-shell` (runs the `code` CLI precisely on the project root) over `tauri-plugin-opener`
(OS-default only):

- `Cargo.toml`: `tauri-plugin-shell = "2"`; `lib.rs`: `.plugin(tauri_plugin_shell::init())` beside the
  existing dialog plugin (`lib.rs:523`); `package.json`: `@tauri-apps/plugin-shell`.
- `capabilities/default.json`: add a **tightly scoped** `shell:allow-execute` for the `code` command with
  the project-root argument (a broad shell scope is a sandbox-escape risk).
- UI: an **"Open in VS Code"** action in `ProjectMenu.tsx` / `MenuBar.tsx` (and optionally a button in
  `ProjectStartupModal.tsx` after create) running `code <store.project.root>`. Fail gracefully when `code`
  is not on `PATH` (toast; the viewport host never needs VS Code — editor-only).

## e2e + verification

- Engine scaffold (`tests/e2e`): `new-project` → `get-project` → assert `fs.existsSync(root + "/src/example.lua")`

## Risks

- `createProject` resets the scene + clears the catalog before scaffolding; the `src/example.lua` write runs
  on a fresh project only — existing projects rely on the `loadProject` ensure.
- The two appdata roots must agree: Rust `app_data_dir()=repo_root()/appdata` (`lib.rs:106`) and the
  engine's `appDataRoot()` from `SAFFRON_APPDATA_DIR` (`lib.rs:217`, `assets.cppm:70-80`).
- `shell` capability scope is security-sensitive — allow only `code` + the project-root arg; do not grant a
  broad execute scope. `code` may be absent (toolbox/headless) — degrade gracefully.
- The Inspector array-of-slots UI is more involved than scalar component editors; reuse the existing
  array/material-set rendering pattern rather than inventing one.
- If a configurable entry path or scripts-as-catalog-assets is later wanted, that requires `project.json` /
  catalog changes + serde regen — explicitly out of MVP scope.
