# scripting MVP plan

This plan adds **per-entity Lua 5.5 scripting** to the engine. An entity carries a **`ScriptComponent`** —
an *ordered list of script slots* — and each slot references a `.lua` file that **returns a class table**
with an `on_update(self, dt)` method. When the editor enters **Play**, the engine instantiates each slot
once per owning entity (binding `self.entity` to that entity), and calls `self:on_update(dt)` every frame.
From inside it a script reaches its own and other entities (by name), reads their components, and moves
entities and the camera. Each script **declares its own editable fields** (a `properties` table); the
editor renders them in the Inspector and stores per-instance **overrides** in the scene.

The defining constraint is **separation**. All knowledge of Lua lives in one new package,
**`Saffron.Script`**. The `ScriptComponent` added to `Saffron.Scene` is **dumb data** — a list of
`{ scriptPath, overrides }` — with zero Lua knowledge. The VM, the bindings, and the edit-time
schema-reader all live inside `Saffron.Script`, and the **Host** is the single place that owns the VM
and bridges it to the existing play seams. No other module (`SceneEdit`, `Rendering`, `Control`, `App`)
gains a dependency on Lua, LuaBridge3, or the bindings.

> **Scope note.** The original ask was a single global `on_update`. Through design discussion it became
> the **Unity-flavored per-entity model**: per-entity script components, multiple ordered script slots
> per entity, `self.entity`, and script-declared editable fields. That is what this plan now targets. It
> is larger than the original MVP by deliberate choice; each phase remains independently shippable.

## Why this is mostly glue, not new machinery

The engine already owns every seam this needs; they are present, documented, and **unused**:

- **A play/simulate state machine.** `PlayState {Edit, Playing, Paused}` on `SceneEditContext`
  (`scene_edit_context.cppm:147`). `enterPlay` *duplicates* the authored scene into `ctx.playScene` **by
  JSON serde** (`scene_edit_play.cpp:82`); all access goes through `activeScene(ctx)`
  (`scene_edit_context.cppm:198`); `stopPlay` **discards** the duplicate — *"the discard is the restore"*
  (`scene_edit_play.cpp:150`). Because the play scene is built by serde, **a registered `ScriptComponent`
  rides into it for free** and any runtime mutation is discarded on Stop.
- **A purpose-built, empty simulation tick.** `tickPlay(ctx, dt)` (`scene_edit_play.cpp:177-197`) gates on
  play state, honors single-step, fixes the step to `1/60`, clamps `dt`, and ends in
  `static_cast<void>(dt)` under the comment *"the simulation seam: physics, scripting, and animation
  advance `*ctx.playScene` here."*
- **A purpose-built, zero-subscriber lifecycle signal.** `SceneEditContext::onPlayStateChanged`
  (`scene_edit_context.cppm:193`) — the create-VM-on-play / destroy-VM-on-stop hook, free to claim.
- **The component machinery a `ScriptComponent` needs.** A component holding a vector of nested structs is
  precedented by `MaterialSetComponent { std::vector<MaterialSlot> }` (`scene.cppm:109,124`); component
  serde is generated by `tools/gen-control-dto/gen.ts` into `scene_component_serde.generated.cpp`;
  components are registered in `registerBuiltinComponents` (`host.cppm:426`) and surfaced to the editor's
  Inspector over the control plane.
- **The opaque-handle + free-function facade for `self.entity`.** `Entity { entt::entity handle }`
  (`scene.cppm:342`); `getComponent/hasComponent/forEach(Scene&, Entity, …)` (`scene.cppm:355`);
  `RelationshipComponent` (parent/children, `scene.cppm:51`) for later hierarchy navigation; the
  type-erased `ComponentRegistry` serialize closures (`scene.cppm:908`) for generic component reads.
- **A per-frame `Layer.onUpdate(TimeSpan)`** (`app.cppm:17`) with the Host owning the loop + the single
  `SceneEditContext` (`host.cppm:407,529`).

What is missing is only: the Lua runtime, the `ScriptComponent` data + a small per-entity instancing
runtime, a tiny binding surface, the edit-time schema reader, and the editor Inspector UI.

## What "done" looks like

- A new **`Saffron.Script`** module wrapping **PUC-Lua 5.5.0** + **LuaBridge3** (both vendored), exposing
  a small `se::` C++ facade and **no Lua symbols** to the rest of the engine.
- An entity can hold a **`ScriptComponent`** = an *ordered* `std::vector<ScriptSlot>`; each
  `ScriptSlot` = `{ scriptPath (relative to the project `src/`), overrides }`.
- A script file **returns a class table** with `on_update(self, dt)` (and optional `on_create`/
  `on_destroy`). On **Play** the runtime instantiates each slot per owning entity — `self = { entity =
  <handle>, …injected fields }` — and calls `self:on_update(dt)` each frame; on **Stop** the instances and
  VM are destroyed. **Within an entity, slots run in list order**; **across entities the order is
  unspecified** (deferred).
- From Lua: `self.entity` (the owning entity), `se.get_entity_by_name(name)`,
  `entity:get_component("Transform")` → table, `entity:set_position/​set_rotation(radians)/set_scale`
  (= move entity), and `se.primary_camera()` + the same setters (= move camera).
- A script **declares editable fields** via a `properties` table; the engine reads the schema at edit time
  in a sandboxed scratch VM; the editor renders the fields in the **Inspector**; per-instance **overrides**
  are stored on the slot and merged into `self` at instance creation.
- A script error **never crashes the host**: each instance's `on_update` runs under `lua_pcall` + a
  `luaL_traceback` handler; on failure the traceback is logged, surfaced to the editor, and play flips to
  **Paused** (the VM is kept).
- **New Project** scaffolds `src/` + a starter example script; the editor offers **Open in VS Code**.
- Scriptable from `se`, covered by `tests/e2e/script.test.ts`, documented under `docs/content/`.

## The caveat that shapes Phase 1: an unverified toolchain combo

No upstream CI matrix confirms **LuaBridge3 building under Clang 21 + libc++ + C++26 + named modules**.
It is header-only, `-fno-exceptions`-clean, honors a consumer-set standard, and CI-tests against Lua
5.5.0 — low risk, but real. **Phase 1 is a deliberate spike:** its first green `make engine` *is* the
verification. The fallback is the raw Lua 5.5 C API behind the same `se::` facade (a one-package swap, no
callers affected). Do not build Phases 2–5 until Phase 1 is green.

## Design decisions (locked)

- **Binding: LuaBridge3 (pin a `3.0-rcN` tag) over vendored PUC-Lua 5.5.0.** sol2 is excluded (hard-`#error`s
  on 5.5; fix PR unmerged). Raw C API is the documented fallback. **Vendor Lua** — CMake 3.31's `FindLua`
  predates 5.5.
- **Module shape:** `Saffron.Script` wraps `LuaBridge.h` + the `extern "C"` Lua headers in the global
  module fragment with **classic `#include`, NO `import std`** (heavy-C++-header rule, like
  `rendering`/`scene`). DAG edge **`Script → {Core, Scene}`**; it depends on neither SceneEdit, Rendering,
  nor Control. The **Host** is the only TU importing both `Script` and `SceneEdit`.
- **Component model: `ScriptComponent { std::vector<ScriptSlot> scripts; }`**, mirroring
  `MaterialSetComponent`/`MaterialSlot` (`scene.cppm:109,124`). **`ScriptSlot { std::string scriptPath;
  Json overrides; }`** — data-only, no Lua. entt keys components by type (one per entity), so *multiple
  scripts per entity* is the **vector**, not multiple components.
- **Script shape: a file returns a class table.** The runtime builds, per slot per entity,
  `self = setmetatable({ entity = <handle> }, { __index = Class })`, injects the merged field values, and
  calls `self:on_update(dt)`. `on_update` is required; `on_create(self)`/`on_destroy(self)` run if present.
- **`self.entity` is the owning entity** (an opaque `ScriptEntity` handle); from it the `ScriptEntity`
  methods reach the scene. Lua never holds an engine pointer.
- **Execution order: within an entity = slot list order (deterministic, free);** across entities =
  **unspecified** (entt iteration order), explicitly deferred. A future `priority` is a sort, not a
  redesign.
- **Editable fields: the script declares them** (`properties` table). The engine reads the schema at edit
  time in a **sandboxed scratch VM**; types are inferred from the defaults (number/bool/string/vec3);
  the editor renders widgets; per-instance **overrides** live on the slot and merge over the defaults into
  `self`. Defaults live in the `.lua`; only overrides live in the scene.
- **Tick seam + lifecycle:** a `std::function<void(Scene&, f32)> simTick` member on `SceneEditContext`
  (a `std`-only type — no Lua leaks) invoked inside `tickPlay`; VM create/destroy ride
  `onPlayStateChanged`. The Host owns the VM and wires both.
- **Scope of effect:** scripts mutate `activeScene(ctx)` (the play duplicate); discarded on Stop. No
  write-back / persistence in the MVP.
- **Camera:** "move camera" = move the scene's primary `CameraComponent` entity's transform
  (`primaryCamera`, `scene.cppm:748`) — never the editor fly-cam.
- **Errors:** per-instance `lua_pcall` + traceback; on error → log, surface to editor, flip play to Paused,
  keep the VM.
- **Sandboxing:** minimal lib set from day one via `luaL_openselectedlibs`
  (base/string/math/table/utf8; withhold io/os/debug/package).
- **Script reference:** a relative path under the project `src/` (e.g. `"turret.lua"`), **not** an asset
  catalog entry in the MVP.

## Explicitly OUT of the MVP (deferred)

Hot reload (manual full-VM rebuild on play-restart only), coroutine scheduler / `wait(seconds)`,
**cross-entity execution-order determinism** (a `priority` field), richer property types (entity/asset/
enum refs, ranges, the descriptor form), the rest of the callback matrix (`on_input`/`on_message`/
`fixed_update`/`late_update`), state write-back / persistence, scripts as catalog assets, a real sandbox,
and LuaJIT. Each extends the same seams.

## Reuse vs build

**Reuse:** the play state machine + `activeScene`/`playScene` discard-on-stop + the JSON serde that carries
the `ScriptComponent` into the play duplicate; the `tickPlay` gating/fixed-step/clamp; `onPlayStateChanged`;
the `MaterialSetComponent`/`MaterialSlot` nested-vector-component pattern; `registerBuiltinComponents` +
the `gen.ts` component-serde codegen; the `Entity` handle + facade + `ComponentRegistry` serialize closures
+ `primaryCamera`; the `registerCommand` + DTO codegen pipeline; the `drain-alarms` cursor pattern; the
Inspector's component-over-control-plane rendering; the e2e harness; the engine-side `createProject`/
`loadProject` scaffold path.

**Build new:** the `lua_static` + LuaBridge3 FetchContent targets; the `Saffron.Script` module (`ScriptVm`
RAII, error→`Result`, the per-entity instancing runtime, the `se`/`ScriptEntity` bindings, the edit-time
schema reader); the `ScriptComponent`/`ScriptSlot` data + serde + registration; the `simTick` hook + Host
wiring; the script control commands + `se` formatter; the Inspector script-slot UI + schema-driven field
widgets; the `src/` scaffold + Open-in-VS-Code; the e2e + docs.

## Status convention

Each phase file carries a `**Status:**` line (`NOT STARTED` / `IN PROGRESS` / `COMPLETED`). Mark a phase
`COMPLETED` only when validation-clean (`make engine` + `make prepare-for-commit`; editor phases also
`bun run check`; wire phases also `make e2e`); delete a phase file only *after* it is `COMPLETED` and
merged. Per the root `AGENTS.md` "keep current" rule, each phase ends by adding/updating its
`docs/content/` page, its e2e coverage, and its `se`/control command in the same change. Adding the module
+ component also means updating the module DAG in `AGENTS.md`.

## Sequencing

Strictly dependency-ordered, lowest-risk first. **1** de-risks the toolchain in isolation (no engine code
imports it) — do not proceed until green. **2** lands the component + the per-entity runtime + lifecycle +
error handling with a minimal `set_position`, so a slot can visibly move its entity (its e2e authors the
component + a test `.lua`). **3** fills out what scripts can read/do (component reads, camera, cross-entity
lookup). **4** adds the script-declared fields end-to-end through the control plane (engine + command,
no UI yet). **5** is the editor surface (Inspector slots + field widgets) plus the authoring UX (scaffold
+ jump-to-code). A reasonable first *visible* slice is **1 + 2 + the `set_position` half of 3** — a cube
with a `ScriptComponent` slot whose `on_update` moves it while playing. Outside `Saffron.Script` the only
edits are the data-only `ScriptComponent` in `Saffron.Scene`, one `std::function` field on
`SceneEditContext`, the four-line Host bridge, additive control commands, and (Phase 5) the Inspector UI +
scaffold.
